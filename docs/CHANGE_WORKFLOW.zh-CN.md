# 变更流程（文档先行，强制执行）

最后更新：2026-03-17

---

## 1. 适用范围

以下改动必须走本流程：
- 数据源接入/移除
- 模型概率/融合/EV/决策逻辑
- 结算时间与切盘规则
- 严格模式（Strict Gate）规则
- 任何会影响 `BUY/WATCH/PASS` 输出的改动

---

## 2. 唯一真相源

改动前先读：
1. `docs/PRODUCT_SPEC.zh-CN.md`
2. 本文件 `docs/CHANGE_WORKFLOW.zh-CN.md`

如与口头描述冲突，以产品规范为准；需要变更规范时，先改规范再改代码。

---

## 3. 标准流程（必须按顺序）

1. **提出变更点**
- 写清楚“为什么改、改什么、影响哪些面板/字段”

2. **更新文档**
- 先更新 `PRODUCT_SPEC.zh-CN.md`
- 若涉及流程，更新本文件

3. **影响面检查**
- 列出受影响模块（数据抓取/融合/模型/决策/UI/定时任务）
- 禁止只改单点导致前后不一致

4. **代码实现**
- 仅按文档实现，不做隐式“想当然 fallback”
- 禁止注入假数据补齐

5. **本地验证（必跑）**
- `npm run job:once -- all`
- `npm run test`
- `npm run build`

6. **结果核对**
- 核对“模型面板 vs 交易决策面板”关键字段一致
- 核对 target_date、weather_target_date、settlement_time 三者口径一致
- 核对 strict gate 阻断原因可解释

7. **提交说明**
- 说明：文档改了什么、代码改了什么、验证结果

---

## 4. 禁止事项

- 未更新文档直接改核心逻辑
- 数据源失败时用假值顶替
- 只修 UI 文案不修底层口径
- 改结算规则但不全链路同步（fetch/model/decision/UI）

---

## 5. 快速检查清单（每次改完都要过）

- [ ] 是否先更新了产品文档
- [ ] 是否明确 sourceKind 与 strict 规则
- [ ] 是否仍以 ZSPD 结算口径为锚
- [ ] 是否禁止假数据 fallback
- [ ] 是否跑了 `job:once -- all` / `test` / `build`
- [ ] 是否检查了页面核心字段一致性
